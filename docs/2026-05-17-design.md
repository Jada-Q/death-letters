# Death Letters — Design Spec

> 2026-05-17 · brainstorm output ready for implementation plan

## 1. 项目一句话

每周 1 封"今晚我可能死去"的我，写给"5 岁的真实 Jada"的信。Claude 作为双人对话的另一方共同产出。52 周完成 1 部公开文学作品。

## 2. 项目本质

- 不是 journaling — 是公开文学实践
- 不是 prompt engineering 示范 — 是 Claude 作为文学共创者的长程实验
- 不是疗愈 — 是 memento mori 仪式
- 不是技术产品 — 是文学产物 + 最小工具支撑

## 3. Brainstorm 确定的 8 个维度

| 维度 | 决定 | 备注 |
|---|---|---|
| 受众 | 公开作品 | 一开始就公开，作者自我审查 |
| 主导 | 双人对话产出 | Claude 是 collaborator 不是 ghostwriter |
| 节奏 | 每周 1 封 × 52 周 | 周日晚固定时间 |
| Form | 演化结构 13 周四季 | Q1 意外 / Q2 选择 / Q3 遗忘 / Q4 完成 |
| 语言 | 中文为主 | 每季末 1 封日译 + 英译（共 4×2 = 8 封翻译） |
| 童年本体 | 真实 Jada 5 岁 | 需 anonymization gate |
| 对话媒介 | 语音 dump + Claude 文字追问 | whisper.cpp 本地转写 + Claude API |
| Publishing | Substack 为主 + GitHub 镜像 | GitHub 是 source of truth |

## 4. Architecture（仓库 + 数据分离）

```
~/Desktop/Projects/death-letters/         (public, GitHub 镜像源)
├── letters/
│   ├── q1-accident/NN-<slug>.md
│   ├── q2-choice/NN-<slug>.md
│   ├── q3-oblivion/NN-<slug>.md
│   └── q4-completion/NN-<slug>.md
├── translations/
│   ├── q1-end-ja.md
│   ├── q1-end-en.md
│   ├── ... (q2/q3/q4 同款)
├── scripts/
│   ├── dialogue.mjs           # voice→transcript→Claude 追问→letter draft
│   ├── anonymize-check.mjs    # commit gate (硬黑名单 + 软警告)
│   ├── translate.mjs          # 季末自动触发，每季 1 封 × 2 语
│   └── publish-helper.mjs     # 复制 markdown 到剪贴板供粘贴 Substack
├── docs/
│   ├── 2026-05-17-design.md   # 本文件
│   └── ANONYMIZE-NOTES.md     # 公开版（不含真名）的匿名化原则说明
└── README.md

~/.death-letters-private/                  (本地 only, 严禁入任何 git)
├── dialogues/YYYY-WW-raw.md              # 含真实童年原文的对话归档
├── voice/YYYY-WW.m4a                     # Voice Memo 原音频
└── ANONYMIZE-CHECKLIST.md                # 真实人名 / 出生地 / 学校名等硬黑名单
```

**核心边界**：含真实童年隐私元素的素材永不进 public repo。

## 5. Weekly Ritual（周日晚 30 分钟，5 步）

| 步 | 动作 | 工具 | 时间 |
|---|---|---|---|
| 1 | 录"今晚我可能怎么死 + 一个童年闪回" | macOS Voice Memos | 3 分钟 |
| 2 | `node scripts/dialogue.mjs` 转写 + Claude 追问 | whisper.cpp + Claude API | 5 分钟 |
| 3 | 回答 3-5 个追问（CLI 内 readline 交互） | terminal | 10 分钟 |
| 4 | Claude 生成 letter draft → 你 Edit 修改 | terminal + editor | 8 分钟 |
| 5 | 跑 anonymize-check → commit → 复制到 Substack 发布 | git + browser | 4 分钟 |

### dialogue.mjs 输入输出

```
INPUT:
  - ~/.death-letters-private/voice/2026-W21.m4a
  - quarter frame（自动按 week 算）
  - 上 1 周 letter 全文（作为对话延续 context）

INTERMEDIATE:
  - ~/.death-letters-private/dialogues/2026-W21-raw.md
    (transcript + Claude 追问 + 你的回答原文)

OUTPUT:
  - letters/<quarter>/NN-<slug>.md
    (final letter draft, 待 Edit 微调)
```

## 6. Letter Spec

```markdown
---
week: 21
quarter: q1-accident
quarter_week: 8
date: 2026-05-24
slug: <kebab-case>
death_mode: <本周想象的死法，一句话>
childhood_anchor: <本周触发的童年意象，一句话>
translations: []   # 季末追加，如 ["ja", "en"]
---

# <title>

亲爱的 5 岁的我，

<正文。无强制小节结构，靠 quarter frame 在 Claude 追问阶段隐性约束。>

—— 即将死去的我，<YYYY 年 W21>
```

frontmatter 字段都是 archive 检索 + 翻译流程 + Substack export 必要项。死法 + 童年锚点是日后回看的可检索面。

## 7. 13 周四季 Frame

```
Q1 (Week 1-13)  死于意外     不可控的死亡。对童年讲"世界会有突袭"。
Q2 (Week 14-26) 死于选择     主动告别。对童年讲"哪些事长大后会主动放下"。
Q3 (Week 27-39) 死于遗忘     慢性消逝。对童年讲"哪些以为永远的都会淡"。
Q4 (Week 40-52) 死于完成     圆满死亡。对童年讲"什么是值得活到底的"。
```

### Quarter Frame 的注入方式

不是硬模板。是 `dialogue.mjs` 在调 Claude API 时，根据当前 week 算出 quarter，把对应的 system prompt fragment 注入到追问阶段：

- 第 8 周 (Q1 中段)：追问偏向"如果今晚一场意外..."
- 第 22 周 (Q2 中段)：追问偏向"如果你今晚选择主动..."
- 第 35 周 (Q3 中段)：追问偏向"如果你正在慢慢消失..."
- 第 48 周 (Q4 中段)：追问偏向"如果今晚就是完成..."

### 季末（W13 / W26 / W39 / W52）特殊处理

季末那封是 **总结性的信**，自动触发 translate.mjs 生成日译 + 英译，存到 `translations/`。

## 8. Anonymization Gate（隐私防线，三层）

这是 _public + 真实童年_ 组合的唯一防失误机制。

### Layer 1 — 硬黑名单（commit fail）

- `~/.death-letters-private/ANONYMIZE-CHECKLIST.md` 列：
  - 你 _真实_ 的家人姓名（父 / 母 / 兄弟姐妹 / 配偶 / 子女）
  - 你的出生地名 / 出生省名 / 童年所在城市
  - 你的童年学校名（小学 / 幼儿园）
  - 童年好友姓名（前 5-10 个常被回忆到的）
  - 你曾用名 / 小名
- `anonymize-check.mjs` 在 commit 前对 staged `letters/**/*.md` 做 case-insensitive grep
- 命中任一条 → 退出码非零 → pre-commit hook 拦下
- checklist 本身在 private 目录，**永不入 public repo**

### Layer 2 — 软警告（commit 通过但 warn）

- 警告条件：
  - 年龄具体数字（"7 岁那年"等可能识别身份）
  - 具体地点（"XX 路 XX 号" / 具体邮编）
  - 时间戳过密（年月日精确到日）
- 你看到 warn 决定要不要改

### Layer 3 — 季末人工反查（不自动化）

每季末（W13 / W26 / W39 / W52）publish 后，你人工 Google：
- `"<letter 标题>" + "Jada"`
- `"<letter 标题>" + "中国"`

看会不会反搜到你的身份。这步不自动化（搜索引擎不允许脚本绕反爬）。

### 失误响应

如果 publish 后发现身份被反推：
1. Substack post 改 unlist（不删，避免好奇心反查）
2. GitHub 对应 letter rewrite，commit 标 `redact: <slug>`
3. ANONYMIZE-CHECKLIST.md 加入新条目
4. 下一周 letter 不致歉，正常推进 — 死亡邮筒的语境本就不需要解释

## 9. Translation Cadence

- 触发：W13 / W26 / W39 / W52 季末信 commit 后，`scripts/translate.mjs` 调 Claude API
- 每季 1 封 × 2 语 = 全年 8 封翻译
- Claude prompt 要点：
  - 保持文学性，不直译
  - 译注（如"中阴期"在英语版加 footnote 注解）
  - ja 版用现代日本文学语体（不用古风），en 版偏 literary essay 不偏 commercial
- 你人工审一遍（30 分钟/季）→ commit → Substack 单独发一篇

## 10. Publishing Flow

GitHub 是 source of truth，Substack 是 distribution。

```
1. git add letters/<quarter>/NN-<slug>.md README.md  # 显式 add，禁 -A
2. git commit -m "week NN: <英文 slug>"
3. git push                                          # 必先跑 anonymize-check pre-commit
4. node scripts/publish-helper.mjs <slug>            # 把 letter markdown 复制到剪贴板
5. 浏览器 Substack New Post → 粘贴 → 检查 frontmatter 没漏出 → publish
```

**不做 Substack API auto-publish**。理由：
- 每周一次的手动粘贴 = ritual 的一部分
- Substack markdown rendering 偶有惊喜（emoji / 块引用 / 表格），肉眼过一遍
- N=52 量级 ROI：自动化省的时间 < 维护 pipeline 的时间
- vendor lock-in 低（迁移 Buttondown / Ghost 只需重新粘贴）

## 11. Edge Cases / Failure Modes

| 故障 | 处理 |
|---|---|
| 周日忘录 | 周一晚补 |
| 连续错过 2 周 | publish "缺席信"（一段说明 + 一句残篇），不假装写满 |
| 录音质量差 transcript 错 | `dialogue.mjs` 显示 transcript 给你 confirm 才进 Claude |
| Anonymize-check 误杀 | 黑名单条目支持 `allow_in_quote: true`，可白名单引用块 |
| Substack 倒闭 | GitHub 是 source of truth，迁移到其他平台只需重新粘贴 |
| 翻译质量不够 | 季末翻译先 commit 到 `draft/q1-end-ja` branch，至少过 7 天再 merge main |
| Claude API 不可用 | `dialogue.mjs` 降级为"只显示追问不生成 letter"，你自己写 |
| 公开后被识别身份 | 见 8.失误响应 |
| 1 年后想停 | Q4 W52 写一封 "完成信" 自然收束，不开 Q5 |

## 12. Out of Scope（明确不做）

- ❌ Mobile app（你不需要）
- ❌ 评论 / 互动系统（Substack 已有）
- ❌ 付费墙（这是作品不是商品）
- ❌ 推荐系统 / SEO 优化（公开但不追读者）
- ❌ AI 自动 publish（违背 ritual 本质）
- ❌ 多个 5 岁人格切换（已选"真实 Jada 5 岁"）
- ❌ 实体 zine（季末 PDF 是 future scope，v1 不做）

## 13. Open Questions（实施前定）

1. **scripts/dialogue.mjs 用 whisper.cpp 本地还是 OpenAI Whisper API？** 本地隐私好但慢；API 快但语音上云。倾向本地。
2. **Anonymize checklist 第一版条目数量？** 估 15-25 条够（家人 + 出生地 + 学校 + 童年好友 5-8 个 + 曾用名）。
3. **Q1 第 1 封什么时候开始？** 推荐 2026-05-24（本周日）= 项目 W1。给自己 1 周建好工具。
4. **季末翻译 draft branch 还是直接写主 branch？** 倾向 draft + 7 天 cooldown。
5. **Substack 发布前是否预告/介绍这个项目？** 倾向不预告，第 1 封信本身就是介绍。

## 14. Success Criteria（成败如何判断）

- **完成成功**：52 周 × 中文信全部 publish + 8 封翻译 publish + 季末反查 0 次身份泄露
- **过程成功**：每周 30 分钟 ritual 真实执行（允许 ≤4 周缺席补漏），不变成应付差事
- **作品成功**：1 年后挑出 7-10 封自认为最好的，编成 PDF zine（v2 scope）
- **kill criteria**：
  - 任何一封信导致家人 / 朋友 / 同事认出并反馈"看了不舒服" → 暂停，复评隐私边界
  - 连续 4 周缺席 → 项目自动 sunset，不强行复活
  - 6 个月后回看前 26 封觉得 ≥1/3 是应付 → 整改 Q3-Q4 frame 或提前 kill
